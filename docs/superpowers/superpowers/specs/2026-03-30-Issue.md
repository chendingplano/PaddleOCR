# Root Cause Analysis: Why the 10x Target Was Missed
## Observed vs. Expected

| Before | After (actual) |After (design target) |
|:-------|:---------------|:---------------------|
| Total OCR gen	| 74,814ms	| 41,353ms	| ~5,776ms |
| Avg per block	| 2,993ms	| 1,654ms	| ~250ms |
| Speedup	| 1×	| 1.81×	| ~10× |
-----------------------------------------

### Root Cause 1 (Primary): The "8–12×" MPS estimate was based on CUDA assumptions that don't hold on Apple Silicon

The design doc's Section 6 estimate assumed:

`MPS speedup: 8–12× for gen`

This figure comes from the general rule that GPU >> CPU for neural network inference. That rule holds on CUDA hardware because dedicated VRAM has ~700 GB/s bandwidth vs. ~50 GB/s for CPU RAM — a ~10–14× memory bandwidth advantage.

On Apple M4, the reality is fundamentally different:

```text
M4 CPU memory bandwidth: 68 GB/s (unified memory)
M4 GPU memory bandwidth: 68 GB/s (same physical bus)
→ Memory bandwidth advantage of MPS over CPU: ~1.0× (none)
```

At `batch_size=1` (the only mode PaddleOCR-VL-0.9B supports — hardcoded in `predictor.py:70–75`), transformer inference is memory-bandwidth bound: each decode step must load all 3.6 GB of fp32 weights through the same memory bus. The M4 GPU and CPU compete for the same 68 GB/s pipe, so moving the model to MPS provides no memory bandwidth advantage.

The 8–12× estimate was derived from CUDA GPU benchmarks (dedicated HBM, much higher bandwidth) incorrectly applied to M4's unified memory architecture.

-----

### Root Cause 2 (Secondary): The observed 1.81× is explained entirely by the dynamic pixels hook — MPS contributes nothing measurable

Quantitative reconstruction:

- Before: 2,993ms at 576 tokens (prefill ≈ 75%, decode ≈ 25%)
- DynPx reduces prefill tokens: 576 → 256 (2.25× fewer)
- Prefill cost scales roughly linearly with token count (linear layers dominate at small T)
- Decode time is unchanged (same ~25 output chars)

```text
Estimated after DynPx alone on CPU:
  prefill: 2245ms × (256/576) = ~998ms
  decode: 748ms (unchanged)
  total: ~1746ms

Actual after (MPS + DynPx): 1654ms
```

The 1,654ms matches the "DynPx alone on CPU" estimate (1,746ms ± normal variance). There is no additional speedup attributable to MPS. If MPS were delivering even 3× for the prefill portion, total would be ~1,077ms — not 1,654ms.

What the implementation does right and wrong
| Component	| Status	| Evidence |
|:----------|:----------|:---------|
| _install_dynamic_pixels_hook	| Working correctly	| 1.81× speedup matches token reduction math |
| _enable_mps_acceleration — .to("mps") call	| Executes without error	| No crash/warning in logs |
| _enable_mps_acceleration — actual inference speedup	| Not working	| gen times match CPU math, not MPS math |
-----

The monkey-patches themselves are technically correct (instance-dict patching, correct argument passing, proper call chain). The problem is not a code bug — it's a flawed performance assumption in the design.

### Why MPS might still not accelerate even if it runs

Two mechanisms could explain zero MPS speedup even with the model on MPS:

1. Unified memory bandwidth parity (most likely): As shown above, M4 GPU and CPU share the same memory bus. Memory-bandwidth-bound inference at batch_size=1 gets no GPU advantage.
2. MPS op fallback: If PYTORCH_ENABLE_MPS_FALLBACK=1 is set (or auto-enabled), operations unsupported by Metal fall back to CPU silently. The constant MPS↔CPU tensor copies could negate any compute gains. The model's RoPE embedding code already handles this: it substitutes device_type="cpu" for "mps" in torch.autocast contexts (modeling_paddleocr_vl.py:138–142) — a sign the model was not designed for MPS as a first-class target.
3. dtype: The model loads as float32 on macOS (line 64 of predictor.py — is_bfloat16_available(None) returns False because device_type is "cpu", not in the accelerator list). fp32 at 3.6 GB is maximally memory-bandwidth-constrained.

----

### The Gap Explained

```text
Expected: ~10x (based on CUDA GPU bandwidth assumptions applied to M4)
Actual:   ~1.8x (DynPx alone; MPS contributes ~0x due to unified memory)
Gap:      ~5.5x unaccounted for
```

The "Combined Expected Impact" table in Section 6 of the design doc assumed MPS would accelerate the model 8–12× because it conflated:

- CUDA-class GPU (dedicated HBM, 700+ GB/s, batch_size agnostic) with
- Apple M4 GPU (unified memory, 68 GB/s, same as CPU, batch_size=1 memory-bound)

### What would actually achieve 10×

To get near 10× on Apple M4, the path forward needs to address the real bottleneck:

1. Reduce model weight memory accesses — quantization (INT8/INT4) would cut the memory bandwidth requirement 4–8×, which directly translates to faster decode on M4's unified memory. This is the dominant lever.
2. Batch multiple blocks together — the current serial block-by-block processing (batch_size=1) wastes compute. Batching 4–8 OCR blocks at once would make the workload more compute-bound (where MPS TFLOPS actually help).
3. Dynamic pixels is the right direction — already delivering ~1.8×; reducing the cap further (e.g., 112px = 12,544 tokens → 64 visual tokens) or using actual image size as the resolution floor would recover more prefill time.
